# 新エンジン設計案 総合比較評価書 (Engine Design Comparison)

本ドキュメントは、`bqn-ledger` の新しい計算エンジン刷新に向けて、これまでに検討された設計オプション（ [SRC_NEXT_ARCHITECTURE_DESIGN.md](docs/SRC_NEXT_ARCHITECTURE_DESIGN.md) 記載の案を含む）と、簿記・会計の保守性・拡張性を担保するために追加提案された新案を統合し、比較・評価したものです。

---

## 1. 解決すべき本質的な課題

新エンジンの設計にあたり、旧エンジンおよび開発中（`src_next`）の以下の短所を克服する必要があります。

```text
【旧エンジンの短所】                     【開発中 (src_next) の短所】
・巨大ハブ (report_engine) の密結合      ・期間や勘定科目の状態計算の重複
・表示追加のたびにハブ修正が必要          ・難解なBQN配列操作の各ファイルへの分散
・計算と表示の完全結合                   ・文字列比較に頼る脆弱なテスト (Fragile Tests)
```

これらを解決し、**「会計的な整合性の自動検証（保守性）」**と**「セクション追加の独立性（拡張性）」**を両立するアーキテクチャを選定します。

---

## 2. 設計オプション一覧（全9案）

検討対象となる設計オプションを、データ処理の流れと責務の観点から分類します。

### 📋 構造設計アプローチ（コードのレイヤー分割）

#### Option 1: Flat Modules (現 src_next 方式)
各モジュール（ `balances` 等）がそれぞれ独自に `cube` を集計し、計算と表示を1ファイルに混在させるフラットな構成。
* **データフロー**: `TSV ➔ Cube ➔ 個別モジュール (計算+表示混在) ➔ stdout`

#### Option 2: Giant Record Hub (旧エンジン方式)
コア計算（ views ）の出力を巨大ハブ（ `report_engine` ）で一度巨大レコードに集約し、表示層（ sections ）に引き渡す。
* **データフロー**: `TSV ➔ Cube ➔ Views (計算) ➔ report_engine (集約) ➔ Sections (表示) ➔ stdout`

#### Option 3: 4-Layer (Views + Format 完全分離方式)
計算（ views ）と人間向け表示（ format ）を完全に別モジュール（ 1セクション＝2ファイル ）に分け、Orchestrator が順次実行する。
* **データフロー**: `TSV ➔ Cube ➔ views/*.bqn (計算) ➔ fmt/*.bqn (表示) ➔ report.bqn (制御) ➔ stdout`

#### Option 4: C+F 中間案 (1セクション1ファイル・関数分離方式)
※ [SRC_NEXT_ARCHITECTURE_DESIGN.md](docs/SRC_NEXT_ARCHITECTURE_DESIGN.md) 推奨案。
計算（ `Build` ）と表示（ `Format` ）を関数レベルで分けるが、ファイルは分けすぎず、同じ物理ファイルに置く。
* **データフロー**: `TSV ➔ Cube ➔ section.bqn (Build関数の計算 ➔ Format関数の表示) ➔ report.bqn (制御) ➔ stdout`

---

### 📥 データ構造・入力正規化アプローチ (Input Normalization)

#### Option 10: Posting IR (正規化仕訳) & 会計カーネル方式 ★最強の土台・新案
TSVの「1行（from/to/amount）」という入力形式をそのまま計算エンジンの基本単位にするのをやめ、内部で必ず簿記的な「Posting IR（`tx_id`, `account_id`, `layer`, `delta`等の配列）」に正規化する層（会計カーネル）を設ける。
TSVは単なる「人間向けの入力形式（Adapter）」とし、Cube以降の計算エンジンは完全に抽象化された Posting IR のみを読み込む。
* **データフロー**: `TSV ➔ Adapter ➔ Transaction/Posting IR ➔ Validation (貸借一致等検証) ➔ Cube / TBDS ➔ Views ➔ Format`
* **メリット**: BQNのColumnar（列指向）な配列処理と最高に相性が良く、将来複数ポスティング（Ledger形式）を導入しても、計算ロジックが一切壊れない。

---

### ⚖️ 会計データモデリング・アプローチ（簿記・計算のパラダイム）

#### Option 5: 試算表セントリック (Trial Balance Centric) 方式 ★新案
BQNの多次元 `cube` を直接ビューからスライスするのを禁止し、中間に一級オブジェクトとして期間・時点を切り出した**「試算表データセット (TBDS)」**を置き、ビューはTBDSからの単純射影 (Filter/Sum) に徹する。
* **データフロー**: `TSV ➔ Cube ➔ TBDS (貸借整合性検証) ➔ Views (TBDSの射影) ➔ Format ➔ stdout`

#### Option 6: JSON/TSV-First (計算・表示の完全分離) 方式 ★新案
BQNエンジンを人間向け表示（Format）の責務から100%解放し、**純粋な計算と中間構造化データ（JSON / 派生TSV）の出力のみ**を行う。整形表示は Go (TUI) や Bash 等の外装が担当する。
* **データフロー**: `TSV ➔ BQN計算エンジン ➔ 中間データ (JSON/TSV) ➔ 外部UIツール ➔ 表示`

#### Option 7: イベントパイプライン (Event Pipeline & Collectors) 方式 ★新案
固定的な3次元 `cube` を廃止し、すべての仕訳をフラットな「イベントストリーム（1次元配列）」として扱い、BQNの関数合成（Combinators）によってフィルタ・グループ化・集計を行う。
* **データフロー**: `TSV ➔ 1次元仕訳ストリーム ➔ フィルター ➔ グループ化 ➔ 各種コレクター(集約) ➔ 表示`

#### Option 8: 宣言的メタデータ駆動 (Declarative Metadata-Driven) 方式 ★新案
BQN側は汎用クエリ・集計エンジンとして振る舞い、レポートのレイアウトや勘定科目のグループ定義を設定ファイル（ TSV ）の記述のみから動的に構築する。
* **データフロー**: `TSV + レイアウト定義TSV ➔ BQN汎用集計エンジン ➔ 動的レンダリング`

---

### 🔄 その他

#### Option 9: 旧 views 再利用方式
`src_next` の cube から旧エンジン（ `src/views/` ）の計算ロジックをインポートして呼び出し、表示だけを新しく書き直す。
* **データフロー**: `New Cube ➔ 旧 Views (計算) ➔ New Format ➔ stdout`

## 4. 各設計案のトレードオフ分析

### 入力正規化のトレードオフ
* **「TSV直読 (Option 1-4, 9)」 vs 「Posting IR 経由 (Option 10)」**
  * `TSV直読` はパーサーと計算が直結しているためコードは一時的に短く見えますが、「複数ポスティング (Ledger形式の導入)」など入力形式が変わった瞬間に Cube や View の全てが破壊されます。また、貸借一致（ゼロサム）検証をいつ行うかが曖昧になりがちです。
  * `Posting IR` を導入すると、入力と計算の間に「正規化された仕訳の壁」ができます。BQNが最も得意とする Columnar（列指向）な配列処理の恩恵を最大化でき、不正なデータは Cube や計算に回る前に確実に弾かれます。将来TSVフォーマットが変わっても Adapter を追加・修正するだけで、計算ロジックは完全な無傷で済みます。

### 構造設計（レイヤー分割）のトレードオフ
* **「ファイル数を最小にしたい（Flat / C+F中間）」 vs 「役割を明確に分けたい（4-Layer）」**
  * `Flat` や `C+F中間案` はファイル数が少なく（12レポート = 12ファイル）、個人開発ツールとしての見通しは良いですが、計算と表示が結合するため「見た目を変えただけで計算テストが落ちる」問題が残ります。
  * `4-Layer` や `C+F中間案（関数分離）` は、表示と計算の境界を関数レベルで分離するため、テストの頑健性は上がりますが、BQNモジュールの依存管理（ `•Import` ）がやや複雑になります。

### 会計モデリングのトレードオフ
* **「多次元 Cube (Option 1-4)」 vs 「試算表データセット (Option 5)」**
  * `Cube` (Day × Account × Layer) は強力ですが、各ビューでインデックス（日付や勘定ID）を個別に計算する必要があり、BQNコードが極めて難解になります。
  * `試算表 (TBDS)` を導入すると、難しいインデックス計算は TBDS 生成部に完全に隠蔽され、ビューのコードは単純な「勘定フィルター」となるため、BQNコードの保守性が劇的に向上します。
* **「BQNですべて描画する (Option 1-5)」 vs 「JSON-Firstで表示を外部に任せる (Option 6)」**
  * `JSON-First` は、BQNから文字列操作コードを100%排除できるため、BQNコードが最もスッキリし、テストも容易になります。しかし、GoのTUIやWeb UIなどの「表示側プログラム」を別に開発・維持するコストが発生します。
* **「時系列 Cube (Option 5)」 vs 「イベントパイプライン (Option 7)」**
  * `イベントパイプライン` はメモリ効率が最高で、勘定科目数や期間の制限がありませんが、BQNの難解な関数合成（Combinators）をフル活用するため、AIや人間によるコードの読み書き難易度が跳ね上がります。

## 5. 推奨するハイブリッド移行戦略

どれか一つの極端な案を選ぶのではなく、現在の `src_next` の移行状況（Stage 4b プレ pretrial）を踏まえ、**段階的に拡張可能なハイブリッドアプローチ**を推奨します。

```text
【フェーズ 0：Posting IR (正規化仕訳) による強固な土台構築】
・既存のTSVを直接 Cube に食わせるのをやめ、一旦 Posting IR (`tx_id`, `account_idx`, `layer`, `delta`) に変換する Adapter を実装。
・Cube はこの Posting IR 配列のみを読んで構築する形に修正。これによりゼロサム検証や不整合データの排除（Fail Closed）を計算前に行う。
        ↓
【フェーズ 1：C+F中間案 ＋ 試算表 (TBDS) の導入】
・ファイルは 1セクション＝1ファイル (C+F中間) で見通しを維持。
・内部で直接 cube を触らせず、TBDS を経由させて計算(Build) と 表示(Format) を分離。
・整合性チェックと値テストの容易性を確保する。
        ↓
【フェーズ 2：JSON-First への緩やかな移行 (UIの外部化)】
・Build関数の戻り値 (値レコード) を JSON 化して書き出せるエントリーポイントを追加。
・BQNの Format関数を退役させ、Go TUI や Bash 外装での描画に切り替える。
```

### この戦略が優れている理由
1. **完全な関心の分離**: 入力正規化（フェーズ0）、計算ロジック（フェーズ1）、表示（フェーズ2）が明確に切り離され、「見た目をいじったら計算が壊れた」「TSVの列を足したら全部壊れた」という悲劇が根絶されます。
2. **壊さずに進められる**: Posting IR と TSV から同じ Cube が作れることを fixture で保証しながら、`tools/check.sh` のテストを維持したまま安全に 1セクションずつ書き換えることができます。
3. **安全ガード**: Posting IR 生成時点での貸借バランス検証と、TBDS 生成時点での日付・状態検証の二段構えになるため、新旧エンジンの差分比較（Stage 4b）や今後の機能追加における信頼性が最高レベルになります。
4. **将来の拡張性**: いずれ必要になる「複数ポスティング (Ledger式記帳)」や、表示側の「Web UI / Go TUI」化など、どの方向への拡張も「間にAdapterを挟む」「出口をJSONにする」だけで対応可能になります。
