# ROADMAP: BQN家計簿を壊さず育てる

- この文書: 長期の方針（地図）
- `TODO.md`: 直近のタスク管理（チェックリスト）
- `docs/README.md`: ドキュメントの入口（導線）

この文書は、`bqn-ledger` を段階的に育てるための道しるべです。

目的は、既存システムを壊さず、理解できないまま機能が増えて散らからないように、家計管理・複式簿記・ledger互換・確定申告準備を順番に学びながら進めることです。

---

## 0. 現在の位置づけ

`bqn-ledger` は、普通のGUI家計簿アプリではありません。

しかし、家計管理システムとしての骨格はすでにあります。

- `journal.tsv`: 実績の正データ
- `plan.tsv`: 未来予定の正データ
- `budget_alloc.tsv`: 予算配賦の正データ
- `accounts.tsv`: 勘定科目とメタ情報の正データ
- `tools/report`: 人間向けの主要入口（`src_next/report.bqn` を呼ぶ）

このプロジェクトの中心は、**TSVを正データにして、BQNで派生ビューを作ること**です。

---

## 1. 絶対に守る基本方針

### 1.1 TSVを正データにする

`journal.tsv` / `plan.tsv` / `budget_alloc.tsv` / `accounts.tsv` が正データです。

派生ファイルやレポート出力を正データにしてはいけません。

### 1.2 先頭5列を壊さない

journal-like TSV の基本形式は固定します。

```tsv
日付	摘要	From	To	金額
```

6列目以降は拡張メタ情報として使います。

```tsv
2026-06-01	スーパー	assets:smbc	expenses:食費	800	tax=private	biz=0
```

既存の先頭5列の意味を変更してはいけません。

### 1.3 拡張は追加で行う

新しい情報が必要になったら、先頭5列を変えず、6列目以降に `key=value` 形式で足します。

例:

```tsv
txn_id=2026-000123
party=Amazon
receipt=receipts/2026/2026-000123.pdf
tax=private
biz_rate=0.30
```

既存キーの意味を変えるより、新しいキーを追加します。

### 1.4 生活用と税務用を混ぜすぎない

この家計簿の第一目的は、毎日の生活で使えることです。

確定申告向けの機能は重要ですが、生活用の見やすさ・入力しやすさを壊してはいけません。

生活サイクルと税務年度は別物として扱います。

- 生活管理: 次の収入日まで使い切らない
- 税務管理: 暦年や申告区分に沿って集計する

### 1.5 BQN版をledger形式に無理やり変えない

BQN版はTSV一行形式を保ちます。

ledger / hledger との接続が必要になったら、BQN版のデータから ledger 形式を **export** します。

BQN版そのものをledger互換ジャーナルへ置き換えないこと。

### 1.6 時間をラベルではなく座標軸として扱う

日付を単なる行ラベルや並び順に戻さない。

- Eventを用途に応じた時間座標へprojectionする。
- `as_of`はEventの日付ではなく観察者の時点として扱う。
- cycle、月、週は時間座標上の区間viewとして扱う。
- Plan / Actual / Budget / Forecastは、同じ時間座標上の異なる意味Layerとして分離する。
- Residual / Scenarioは、当面Cubeの基本軸ではなく派生観察viewとする。
- 複数時間を扱ってもActualを二重計上しない。

詳細は`docs/TIME_AS_AXIS.md`を参照する。

## 2. AI作業ルール

AIがこのリポジトリを触る場合は、必ずこの章を守ります。

### 2.1 大改造しない

一度に大きなリファクタリングをしないこと。

変更は、原則として1段階・1目的・小さい差分にします。

### 2.2 理解不能な抽象化を入れない

ユーザーが理解できないまま、複雑な抽象化・設定層・自動生成を増やしてはいけません。

便利さより、読めること・直せることを優先します。

### 2.3 正データを勝手に書き換えない

AIは、実データの `journal.tsv` / `plan.tsv` / `budget_alloc.tsv` を勝手に書き換えてはいけません。

入力支援ツールを作る場合も、保存前に生成される行を表示し、確認できる形を優先します。

### 2.4 派生ファイルを正データ化しない

`budget_journal.tsv` や export 結果は派生物です。

必要がある場合だけ生成し、基本的には `journal.tsv` / `accounts.tsv` / `budget_alloc.tsv` から再生成できるようにします。

### 2.5 仕様変更は必ずdocsに書く

TSVの列、メタキー、集計ルール、表示ルールを変える場合は、必ず `docs/` を更新します。

特にメタキーを増やす場合は、`docs/JOURNAL_META.md` と `docs/CONVENTIONS.md` を更新します。

### 2.6 税務判断をコードに埋め込まない

税務制度は変わる可能性があります。

確定申告に関係する実装では、制度上の判断をコードに直書きせず、設定・メタ情報・ドキュメントに分けます。

不明点は「仮」「未確定」と明記します。

### 2.7 まず検査を通す

変更後は、可能なら以下を実行します。

```sh
./tools/check.sh
```

スナップショットを意図的に変える場合だけ、理由を説明して更新します。

```sh
./tools/snapshot.sh --update
```

## 3. 段階的ロードマップ

各段階は、前の段階を壊さずに進めます。

次の段階へ進む前に、ユーザーがその段階の意味を説明できることを目安にします。

---

### Phase 1: 現在のBQN家計簿を安定運用する

関連ドキュメント:
- `docs/MAINTENANCE.md`
- `docs/CONVENTIONS.md`
- `docs/JOURNAL_META.md`
- `docs/CYCLE.md`
- `docs/ARCHITECTURE.md`
- `docs/REPORT_DESIGN.md`

#### 目的

今の生活用家計簿を壊さず、毎日使える状態を保つ。

#### 学ぶこと

- TSVが正データであること
- From / To / Amount の意味
- `accounts.tsv` による科目検証
- 実績と予定と予算配賦の違い

#### やること

- `journal.tsv` を手で読める状態に保つ
- `accounts.tsv` にない科目を増やさない
- `budget_alloc.tsv` を予算配賦の正データとして扱う
- `tools/report` を日常入口にする
- 内部モデルは `day × account × layer` の **Canonical Daily Cube** に移行済み。レポート計算の基盤となっている。

#### やらないこと

- いきなり税務対応を完成させない
- ledger形式へ全面移行しない
- GUI化を急がない
- 自動化でデータ削除をしない

---

### Phase 2: メタ情報の最小セットを決める

関連ドキュメント:
- `docs/JOURNAL_META.md`
- `docs/CONVENTIONS.md`
- `tools/add.bqn`

#### 目的

確定申告やledger exportに備えて、6列目以降に置く基本メタキーを決める。

#### 学ぶこと

- メタ情報は計算本体とは別に置けること
- 先頭5列を固定しながら拡張すること
- 生活用の記録と税務用の記録を分けて考えること

#### 候補キー

```tsv
txn_id=        # 同じ取引・レシートを束ねるID
party=         # 相手先
receipt=       # 領収書・請求書・明細への参照
tax=           # private / business / mixed
biz_rate=      # 事業使用割合。例: 0.30
invoice=       # インボイス等の暫定メモ
note=          # 短い補足
```

#### やること

- `docs/JOURNAL_META.md` にキーの意味を追記する
- `docs/CONVENTIONS.md` に表記ルールを追記する
- `tools/add.bqn` がメタ列を扱えるか確認する

#### やらないこと

- まだ税額計算しない
- まだ申告書を作ろうとしない
- まだ固定資産や減価償却へ進まない

---

### Phase 3: 複数行を1取引として束ねる

#### 目的

ledgerの複数postingに近い考え方を、TSV一行形式を壊さずに扱えるようにする。

#### 学ぶこと

- 1枚のレシートに複数科目が混ざること
- 同じ `txn_id` で複数行を束ねられること
- BQN版の軽さと複式簿記の厚みを両立すること

#### 例

```tsv
2026-06-01	スーパー	assets:smbc	expenses:食費	800	txn_id=2026-0001	party=スーパー
2026-06-01	スーパー	assets:smbc	expenses:日用品	300	txn_id=2026-0001	party=スーパー
```

#### やること

- 同じ `txn_id` の行を一覧できるツールを作る
- `txn_id` 未設定行を検出できるようにするか検討する
- レシート単位で金額を確認できる出力を検討する

#### やらないこと

- 既存の一行一取引入力を禁止しない
- すべての行にいきなり `txn_id` を必須化しない

---

### Phase 4: tax-summaryを作る

#### 目的

確定申告の準備として、事業用・私用・按分を分けた年次集計を出す。

#### 学ぶこと

- 生活サイクルと税務年度は別であること
- `tax=private` は税務集計から除外すること
- `tax=business` は事業用として集計すること
- `tax=mixed` は `biz_rate` で按分すること

#### 想定コマンド

```sh
bqn tools/tax-summary.bqn 2026
```

#### 出したいもの

- 事業収入合計
- 事業経費合計
- 科目別経費
- 私用除外後の金額
- 按分後の金額
- `receipt=` 未設定の一覧
- `tax=` 未設定の一覧

#### やらないこと

- 税額を自動計算しない
- 申告書そのものを作らない
- 税務判断を決め打ちしない

---

### Phase 5: ledger exportを作る

#### 目的

BQN版を正データにしたまま、ledger / hledger で検算できる出口を作る。

#### 学ぶこと

- ledger形式とBQN TSV形式の違い
- exportは派生物であり、正データではないこと
- 外部形式へ逃がせる安心感

#### 想定コマンド

```sh
bqn tools/export-ledger.sh > journal.ledger
```

#### 例

TSV:

```tsv
2026-06-01	スーパー	assets:smbc	expenses:食費	800
```

ledger出力:

```ledger
2026-06-01 スーパー
    expenses:食費        800 JPY
    assets:smbc
```

#### やること

- まずは単純な From / To / Amount からexportする
- `txn_id` が同じ行をまとめるかは後で検討する
- export結果を正データとして編集しない

#### やらないこと

- BQN版のTSVをledger形式へ置き換えない
- hledger依存を本体に戻さない

---

### Phase 6: 残高突合を学ぶ

#### 目的

帳簿の残高と、実際の銀行・決済サービスの残高を照合できるようにする。

#### 学ぶこと

- 記録上の残高と現実の残高を合わせること
- 照合済みと未照合を分けること
- 月末残高や明細との関係

#### 候補メタ

```tsv
cleared=1
statement=2026-06-smbc
```

または別ファイル案:

```tsv
date	account	balance
2026-06-30	assets:smbc	123456
```

#### やること

- `assets:*` の残高確認方法を決める
- 月次で確認できる出力を作る
- `cleared` を必須にするかは急がず検討する

#### やらないこと

- 全行をいきなり照合必須にしない
- 銀行明細インポートを急がない

---

### Phase 7: 帳簿らしい出力を作る

#### 目的

複式簿記として、外から見ても帳簿らしい出力を作る。

#### 学ぶこと

- 仕訳帳
- 総勘定元帳
- 試算表
- 損益計算書
- 貸借対照表

#### 想定ツール

```sh
bqn tools/export-journal-book.bqn
bqn tools/export-general-ledger.bqn
bqn tools/export-trial-balance.bqn
bqn tools/export-profit-loss.bqn
bqn tools/export-balance-sheet.bqn
```

#### やること

- まずTSV出力で作る
- 人間が読めることを優先する
- 必要ならあとでCSVやledger形式へ変換する

#### やらないこと

- 見た目を先に豪華にしない
- 簿記用語だけを増やして意味不明にしない

---

### Phase 8: 証憑保存と履歴を整える

#### 目的

帳簿の数字と、領収書・請求書・明細を結びつける。

#### 学ぶこと

- 証憑とは何か
- 帳簿と証憑をどう結ぶか
- 保存期間や年度管理をどう考えるか

#### 候補構成

```text
receipts/
  2026/
    2026-000123.pdf
    2026-000124.jpg
```

TSV側:

```tsv
receipt=receipts/2026/2026-000123.pdf
```

#### やること

- `receipt=` の命名規則を決める
- 年度別に保存する
- Gitで管理するか、別保管にするか決める

#### やらないこと

- 個人情報を公開リポジトリに入れない
- 本番データとサンプルデータを混ぜない

---

### Phase 9: 固定資産・減価償却を別枠で考える

#### 目的

高額なPC・機材・ソフトなど、日々の支出とは別に扱う必要があるものを整理する。

#### 学ぶこと

- 消耗品と固定資産の違い
- 減価償却という考え方
- 私用・事業用の按分

#### 候補ファイル

```tsv
asset_id	name	acquired	cost	biz_rate	method	note
```

例:

```tsv
asset-2026-001	PC	2026-06-01	200000	0.50	unknown	仮登録
```

#### やること

- まず台帳だけ作る
- 税務処理は急がない
- 必要になった時点で制度確認する

#### やらないこと

- 日々の家計簿本体に無理に混ぜない
- 税務ルールを曖昧なまま自動化しない

## 4. 進め方の合言葉

このプロジェクトでは、次の順番を守ります。

```text
理解する
記録する
検査する
表示する
外へ出す
自動化する
```

自動化は最後です。

理解できないものを先に自動化すると、システムが散らかります。

## 5. 優先順位まとめ

```text
1. 今のBQN家計簿を毎日使う
2. 先頭5列を守る
3. 6列目以降のメタ情報を少しずつ足す
4. txn_id / party / receipt を育てる
5. tax / biz_rate で私用・事業用を分ける
6. tax-summaryで暦年集計を出す
7. ledger exportで外部検算の橋を作る
8. 残高突合を学ぶ
9. 帳簿らしい出力を作る
10. 固定資産・減価償却は必要になってから別枠で扱う
```

## 6. AIへの最終注意

このリポジトリは、機能を増やすことより、ユーザーが生活の中で理解して使えることを重視します。

AIは、次のことをしてはいけません。

- 既存のTSV形式を勝手に変える
- 本番データを書き換える
- ユーザーが理解していない抽象化を増やす
- 税務判断を断定してコード化する
- 派生ファイルを正データとして扱う
- 便利そうだからという理由だけで機能を足す

AIがするべきことは、次のことです。

- 小さく提案する
- 既存の設計を読む
- docsを更新する
- サンプルで確認する
- ユーザーが説明できる粒度に保つ
- 生活用の家計簿として壊れないことを最優先にする
