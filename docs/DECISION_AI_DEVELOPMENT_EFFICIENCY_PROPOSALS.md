# AI開発効率向上およびトークン最適化ツールの提案（2026-06-21）

## 背景と目的

BQN-Ledgerは、「TSVデータ ＋ 配列志向言語（BQN）計算エンジン ＋ Go/Bashツール群」という独自の堅牢な構造を持っています。しかし、AIエージェントが本リポジトリで作業するにあたり、以下の課題から**コンテキスト（トークン）の過剰消費**および**デバッグのターン数（往復）の増加**が発生しています。

1. **レポート全体（160行以上）の毎回の読み込みによるトークン消費**
2. **BQNの抽象的なエラーメッセージに対する、脳内エミュレーションコスト**
3. **TSV間のマッピング（`accounts.tsv` と `journal.tsv` / `plan.tsv` 等）の目視突合コスト**
4. **タブ区切りTSVのテキスト置換における構文崩れの防止コスト**

これらの課題を減らし、AIエージェントの作業精度向上と開発効率化に役立つ具体的な支援ツールを提案します。

---

## 提案ツール一覧

### 1. Output Squeezer (レポート数値のピンポイントクエリツール)

* **概要**: 検証時に `main.bqn` の出力全体をAIコンテキストへ流し込むのを防ぐため、確認したい特定の数値やセクションのみを抽出して出力するCLIコマンド。
* **状態**: historical / removed from current tree (2026-06-26 re-audit)。`tools/sqz-report` は旧エンジン削除後の現在の tree には存在しません。設計記録は `docs/OUTPUT_SQUEEZER_DESIGN.md` に残っていますが、現行確認では `tools/report-next-summary`、`checks/check-src-next-compact-summary.sh`、`rtk`、または `sqz compress` の pipe 利用を使います。復活させる場合は別の tool-only task として扱います。
* **動作イメージ**:
  ```bash
  # 特定のアカウントの残高のみを抽出
  rtk query-report --base fixtures/generalization-moko --date 2026-06-16 --key liquid.SMBC
  # 期待値（Expected）との数値上の差分（diff）のみをJSONで出力
  rtk query-report --diff fixtures/generalization-moko expected/
  ```
* **トークン効果**: 毎回数千トークン消費していたレポート読み込みを、数十〜数百トークン程度に抑えられる場合があります。

### 2. BQN REPL & Variable Dumper (BQNデバッグの効率化)

* **概要**: BQNの型エラーや境界エラー（`𝕨⊑𝕩: 𝕩 must be a list` など）が発生した際、どのローカル変数が異常値（`@` など）になっているかを可視化するツール。
* **構成要素**:
  1. **BQN REPL**: 任意のBQNワンライナーを実行して型や評価結果を確認できるサンドボックス環境。→ `tools/bqn-eval` (Phase 1) として実装済み。
  2. **Variable Dumper**: エラー発生時にそのレキシカルスコープ of 全変数の「名前・型・値」をダンプするデバッグマクロ/ヘルパー。→ `tools/bqn-dump` (Phase 2, 2026-06-27) として実装済み。kind / shape / preview / boxed hint を出力。
* **トークン効果**: ソースコード全体を何度も再読み込みして脳内トレースするコストを減らし、原因特定を早められる可能性があります。

### 3. TSV Alignment Linter (メタデータと実データの静的突合ツール)

* **概要**: `accounts.tsv` と `journal.tsv` / `plan.tsv` / `cycle.tsv` 間の「アカウント名のタイポ」「メタデータ不整合」「紐付け不一致」を静的に検出する高速チェッカー。
* **検出対象**:
  * `journal.tsv` 等で使用されているが、`accounts.tsv` に登録されていないアカウント。
  * アカウントの `budget=` メタデータで指定された封筒が、`role=budget` として定義されていない。
  * `cycle.tsv` の `income_account` に指定されたアカウントが、`role=income` ではない。
* **状態**: 実装完了 (2026-06-22)。`checks/lint_accounts.bqn` および単体テスト `tests/test_lint_accounts.bqn` にて、各アカウントの `role` メタデータ定義の有無や接頭辞（`assets:`, `budget:` 等）との整合性チェック、および `budget=<name>` で指定されたターゲットアカウントが実際に存在し、かつ `role=budget` を持っているかどうかの整合性検証ロジックを実装しました。
* **トークン効果**: 大きなTSVをすべてAIのコンテキストに読み込んで目視確認する必要を減らし、エラーメッセージの数行の指摘から修正方針を立てやすくします。

### 4. Structured TSV Patch Applier (エージェント用構造化データ編集ラッパー)

* **概要**: AIエージェントがTSVを編集（あるいは修正案を作成）する際、テキスト置換ではなく、JSONなどの構造化データを通じて安全にレコードを追加・更新するコマンド。
* **動作イメージ**:
  ```bash
  go-editor patch --file accounts.tsv --action update --account "食費" --key "budget" --val "食費封筒"
  ```
* **トークン効果**: テキストレベルでのDiff作成や置換コード作成にかかるトークンを減らし、タブずれや改行コードの問題による構文破壊リスクを大きく下げられます。

### A. 既存提案の補強・気づき

1. **BQNのエラーダンプの充実（REPL/Dumperの補強）**
   * BQNは実行時エラー（例: `𝕩 cannot be empty` 等）のトレースが抽象的なため、エラー時に**「エラー箇所で評価中だった引数 `𝕨` / `𝕩` の型・値・shape」**を同時に自動ダンプする仕組みがあると、脳内エミュレーションコストを大きく下げられます。
2. **Patch Applier による環境・パーミッション依存の隠蔽**
   * CLI実行時のパーミッション揺らぎ（`os error 13` 等）を Go レベルの PATCH API で包んで隠蔽することで、環境依存のCLI実行失敗を回避し、AIの往復ターン数を減らせる可能性があります。

### B. 新たな提案ツール

### 5. Golden Diff Summary (差分可視化のサマリーフィルター)
* **概要**: ゴールデンテスト（`golden_check.sh`）失敗時に `diff -u` の生結果をすべてAIに読み込ませるとトークンが爆発するため、一定行数を超える差分は自動でサマリー化（不一致の行数と概要の提示）し、必要な部分だけをAIがピンポイントで確認できるようにするフィルター。
* **状態**: 実装完了 (2026-06-22)。`checks/golden_check.sh` 内に `MAX_DIFF_LINES=30` としきい値を設け、それを超える場合は `head` と `tail` を使って差分の先頭・末尾のみをダンプし、中間の行数を省略表示する `show_diff_and_exit` 処理を組み込みました。

### 6. Context Unload / Task-focused Subagents (作業コンテキストの明示的刈り込み)
* **概要**: 会話が長くなることによるコンテキストの肥大化を防ぐため、リポジトリ全体の探索や巨大ファイルの解析は `invoke_subagent` でスパンしたサブエージェントに切り出して実行させ、親エージェントのコンテキストをクリーンに保つ作業ルールの徹底。

### D. 実作業から見つかった設計・検証の課題と改善案

### 10. BQNの同質化（マージ）仕様に起因する型揺らぎの予防ナレッジ（AI更新ガイドライン）
* **意見追記に至った経緯・文脈**:
  `report_engine.bqn` で `section_status_values ⇐ ⟨check_status, actcmp_status⟩` と定義した際、値が同じ長さ（例：`"OK"` と `"OK"` で長さ2）の時は自動的に 2D 配列にマージされ、長さが異なるとき（例：`"WARN"`（4）と `"OK"`（2））はネストされたリストになるという、実行時の値によってデータ型が動的に揺らぐ現象に直面しました。
  これにより、テストコード側で `GetStatus` ヘルパーがインデックス選択（`⊏`）した結果のデータ構造が、あるフィクスチャでは `⟨ "OK" ⟩`（ボックスされたリスト）、別のフィクスチャでは `"WARN"`（文字列）のようになるなど予測不可能な型崩れが発生し、BQNから `Error: Trying to output non-character` などの抽象的な実行エラーが返されました。
  この解決のために、単項 `<` (Enclose) で要素を明示的に保護して `⟨<check_status, <actcmp_status⟩` とし、取り出し側に単項 `>` (Disclose) と `⊑` (First) を組み合わせて適用する設計にたどり着くまでに、何往復もの REPL ワンライナー検証とトークンの消費が発生しました。
* **提案内容**:
  * 提案2の **BQN REPL / Variable Dumper** の早期導入が望まれます。
  * また、BQNの自動マージ（Homogenization）仕様や Enclose (`<`)/Disclose (`>`) などの配列志向特有のハマりどころをまとめた**「AI向けのコーディング規約・チートシート（例: `docs/BQN_CONVENTIONS_FOR_AI.md`）」**を整備しておくことで、AIが初手からより安全なコードを書きやすくなり、エラーデバッグのターン数を減らせます。

### 11. 差分セルフレビューのルーティン化による文脈刈り込み
* **意見追記に至った経緯・文脈**:
  BQNの型エラーやテストの期待値不一致に対処する過程で、巨大なコードファイルやテスト用 TSV データを何度も読み直して脳内シミュレーションを行うと、AIのコンテキスト（トークン）が急速に肥大化する問題がありました。
  しかし、`rtk git diff` で自分が加えた変更の差分だけをピンポイントで確認し、論理的なデータ構造の変化だけに集中してデバッグを行ったところ、自己修正（例：`GetStatus` の取り出し時の Disclose を `> ⊑ idx ⊏ values` に修正する、`empty-journal` の期待値を `ERROR` に修正するなど）が最小限のターンで完了しました。
* **提案内容**:
  AIエージェントがテスト実行前やコミット前に、必ず `git diff` を単一のコマンドで実行し、そこから得られた差分を元に最終整合性をセルフチェックする運用ルールの明示。

### 12. リンター警告によるドキュメント更新漏れの即時検知
* **意見追記に至った経緯・文脈**:
  本実装で `report_engine` の返り値フィールドを3つ追加した際、`docs/REPORT_FIELD_MAP.md` の「総フィールド数」の定義値（96個から99個）やテーブルへの追記が漏れた状態で全体のテストを走らせました。このとき、提案9で導入された `check-docs-drift.sh` が即座に「docs drift detected」と警告を出して異常終了したため、AIは迷いを減らし、人間のレビューを受ける前にその場でドキュメントの同期へ進めました。
* **提案内容**:
  コードとドキュメント（または設定ファイル）の整合性を自動で検証する「乖離検知リンター」を各レイヤーに組み込むことで、AIの作業抜け漏れを早期に検出し、開発品質を保ちやすくできます。

### 14. 例外・エラー系テストの堅牢化（Fragile Test の防止）
* **追記に至った経緯**: 
  全体テストを実行した際、`check-negative.sh` において `bqn-eval` のエラーメッセージの期待値（`expression is required`）と実際のエラー文言（`expression is empty`）が不一致となりテストが落ちました。これは、以前に誰かがツールのエラーメッセージを親切なものに変更した際、テスト側のアサーションを修正し忘れたために起きた事象です。
* **提案内容**: 
  提案7で導入された例外系テストのアサーションにおいて、出力文言の「完全一致」や「長文の具体的なフレーズ」に強く依存しない設計にする。具体的には、終了コード（`status != 0`）の確認を主軸にし、メッセージチェックは「`Error:`」や「`Usage:`」といった抽象的なキーワードのみに留めるか、内部的に構造化されたエラーコード（例: `E_MISSING_EXPR`）を出力・検証する設計にすることで、ツール改善に伴うテストの追従漏れ（Fragile Test化）を防ぎます。

### 15. Output Squeezer の「Impact Summary (影響範囲差分サマリー)」モード（historical / tool removed）
* **追記に至った経緯**: 
  提案1の `sqz-report` は当時、数値のピンポイント抽出やアサーションに有用でした。ただし現在の tree では削除済みです。もし復活させるなら、AIが「既存のロジックやデータを変更した結果、全体の中でどの指標に影響が出たか」を探索的に知るための機能として再設計します。
* **提案内容**: 
  復活させる場合は、変更前と変更後の状態を比較し、「数値に変動があったキーだけを一覧で返す（差分サマリー）」モードを追加する。これにより、AIは「予想外の場所（例えば別の封筒や来月の予測）に影響が出ていないか」を数千トークン消費して全体差分（diff）を読むことなく、変動箇所のみの最小限のコンテキストで即座に検知できるようになります。

2026-06-22の「`src/input/finish.go` の退役および `tools/legacy/finish-preview.go` への移動・検証」の移行作業を通じて、AI作業の精度向上と手戻り防止における、以下の知見とベストプラクティスを得ました。

### F. 実作業から見つかった設計・検証の課題と改善案

### 16. リネーム・移動時の履歴追跡のベストプラクティス化（`git mv` の優先利用）
* **追記に至った経緯**:
  `src/input/finish.go` の移動にあたり、単純な削除（`rm`）と新規作成（`write`）ではなく、`git mv` を用いてリネームしました。これにより、git 履歴上でのファイルの継続性が完全に追跡され、AIが `git status` や `git diff` を実行した際にも「既存ファイルの純粋な移動」であることが一目で判別できました。もし履歴が途切れると、AIはファイル全体を「完全な新規実装」と誤認して全内容をコンテキストに読み込み直す必要が生じ、トークン消費が増加します。
* **提案内容**:
  AIがファイルのディレクトリ移動や名前変更を行う際は、必ず `git mv`（またはそれに相当するツール機能）を優先的に使用することをコーディング規約（`AGENTS.md` 等）に明示する。

### 17. 安全境界（Fail-Safe Boundary / No-Mutation）を検証する動的アサーションの標準化
* **追記に至った経緯**:
  今回の移行作業では、テストスクリプト `check-disabled-features.sh` が、移動後の Go ツールが「確実に preview-only であり、実データ TSV を書き換えないこと（`no-mutation`）」をファイルチェックサムを用いて厳格に検証していました。このような「書き換えを行わないこと（安全境界）」を能動的にアサーションするテストの存在は、AIがリファクタリングを行う上で最大の安心材料となり、手戻りや実データ破損のリスクを劇的に下げました。
* **提案内容**:
  静的解析だけでなく、書き込み権限の境界や副作用の有無を動的にアサーションするネガティブ検証（または No-Mutation 検証）を、重要なスクリプトやツール移動時に必ずセットで維持・作成する設計ルールの推奨。

### 18. 移行・退役計画における「影響範囲とHandoff案」のドキュメントテンプレート化
* **追記に至った経緯**:
  ユーザーが追加した計画書 `LEGACY_FINISH_GO_RETIREMENT_PLAN.md` には、「触る可能性があるファイル」「触ってはいけないファイル」「受け入れ条件」「Handoff 案」が極めて明確に定義されていました。この構造化された指示のおかげで、AIは「どのファイル関係を更新し、何を確認し、何を触ってはならないか」を即座に完璧に把握し、1往復で正確に作業を完了させることができました。
* **提案内容**:
  今後リポジトリ内のリファクタリング、移行、機能退役などの計画書を作成する際は、今回の計画書で採用された以下の要素をテンプレート化し、標準構成とすることを推奨します。
  - **対象・目的・現在地**
  - **移動/修正/維持の判断基準**
  - **触るファイル / 触ってはいけないファイル（境界線）**
  - **移行の受け入れ条件（Acceptance Criteria）**
  - **Handoff 案（AIへの実行指示プロンプトのドラフト）**

### G. 既存ツール（rtk / sqz-report）へのレビュー（historical note）

- **`rtk` (Token-optimized CLI proxy)**:
  `rtk git status` や `rtk git diff` など、AIが実行するCLIコマンドを透過的に最適化してくれる proxy は非常に強力です。特に、差分確認時のノイズ削減やコンテキスト上限への到達防止に直結しており、AIが迷いなくセルフレビューを行うためのインフラとして機能しています。
- **`sqz-report` (Output Squeezer)**:
  当時の設計意図として、レポートの全体出力をAIに読ませることなく、ピンポイントのキーや grep での検証ができる仕組みは有効でした。ただし現在の tree には `tools/sqz-report` は存在しません。現行では `tools/report-next-summary` と `rtk` / `sqz compress` の pipe 利用を前提にします。

## Docs / Implementation Drift Audit 作業に基づく追加フィードバック（2026-06-26 追記）

**経緯**:
2026-06-26 の docs / implementation drift audit では、旧エンジン削除後に残った documentation drift を、`docs/DRIFT_AUDIT_TODO-2026-06-26.md` → `docs/DRIFT_AUDIT-2026-06-26.md` → `docs/DRIFT_FIX_PLAN-2026-06-26.md` の順に分けて処理しました。

この作業を通じて、AI作業精度とトークン効率の向上に効いた点を以下に記録します。

### H. 実作業から見つかった改善案

### 19. Audit TODO → Drift Table → Fix Plan の三段階化

* **気づき**:
  docs drift は、見つけた瞬間に直し始めると文脈が膨らみやすく、古い docs と現行仕様を混ぜやすいです。今回はまず TODO 形式で監査範囲を固定し、次に drift table へ P1/P2 付きで記録し、最後に batch 化した fix plan へ切り出したことで、作業判断がかなり軽くなりました。
* **提案**:
  大きめの docs / implementation drift 監査では、いきなり修正せず、以下を標準形にする。
  1. `*_TODO-YYYY-MM-DD.md` で読む範囲と非目標を固定する。
  2. `*_AUDIT-YYYY-MM-DD.md` で drift table を作る。
  3. `*_FIX_PLAN-YYYY-MM-DD.md` で batch 分割し、完了状況をチェックボックスで管理する。
* **トークン効果**:
  AIが「次にどこを見るか」「これは今直すべきか」を毎ターン再推論しなくてよくなり、会話の後半でも小さい文脈で再開しやすくなります。

### 20. “存在しない guard” を明示的に false confidence として扱う

* **気づき**:
  今回いちばん危険だった drift は、古い docs に `lint_cli.bqn` / `check-forecast-zero` / `section_status_*` などの guard が「実装済み」と残っていたことでした。これは単なる古い説明より危険で、AIが安全確認済みと誤信する原因になります。
* **提案**:
  safety / invariant docs の監査では、古い参照を単に削るだけでなく、`historical / removed / not current guard` と明示する。特に安全規格系 docs では、`GUARDED` を名乗る項目は必ず current tree の file/check/test に対応させる。
* **トークン効果**:
  AIが存在しない check を探し回る時間を減らし、修正判断を「guard の復活」か「docs の降格」へすぐ分岐できます。

### 21. Command wrapper の実測を baseline に含める

* **気づき**:
  ドキュメント上は `sqz <command>` のように前置きできる想定でしたが、実際のローカル `sqz` は stdin/subcommand 型で、`sqz ./tools/check.sh` は失敗しました。一方、`rtk bash ./tools/check.sh` は通りました。
* **提案**:
  監査や handoff の baseline では、チェック本体だけでなく「推奨コマンドが本当にこの環境で動くか」も確認する。失敗した wrapper も baseline issue として記録する。
* **トークン効果**:
  AIが以後の作業で同じ失敗コマンドを繰り返すことを防げます。

### 22. Soft-gated tests は docs drift と同じくらい危険

* **気づき**:
  `tools/check.sh` は「Go editor tests を実行する」と説明されていましたが、実装では `go test` が `|| true` で握りつぶされていました。これは docs と実装のズレであると同時に、AIがチェック結果を過信する原因でした。
* **提案**:
  `tools/check.sh` に含めるテストは原則 gating にする。やむを得ず best-effort にする場合は、コマンド出力と docs の両方に `non-gating` と明記する。
* **トークン効果**:
  AIが「なぜ通っているのに壊れているのか」を後から調査する往復を減らせます。

### 23. Historical docs は削除より “status note + current reading path” が効く

* **気づき**:
  旧エンジン由来の docs は、完全削除すると判断記録を失います。一方で現役入口から current spec として読めると危険です。今回の修正では、historical / removed / current tree には存在しない、という注記が有効でした。
* **提案**:
  大きい historical docs は即削除せず、先頭に status note と current reading path を置く。入口 docs からは active ではなく Done / historical / archive candidate として参照する。
* **トークン効果**:
  AIが古い設計を現行仕様と誤読する確率を下げつつ、必要な背景は残せます。

### 24. Follow-up TODO は audit の最後に必ず残す

* **気づき**:
  audit/fix を一巡すると「まだやること」が複数残ります。会話の終端でそれを自然文だけで返すと、次の pit が再構成に時間を使います。
* **提案**:
  audit 完了時に、残りを `*_FOLLOWUP_TODO-YYYY-MM-DD.md` として残す。各項目に、目的・最初の確認・受け入れ条件・推奨チェックを短く置く。
* **トークン効果**:
  次の pit が最小読み込みで続きから着手でき、ユーザーも「今やる/後でやる」を選びやすくなります。
