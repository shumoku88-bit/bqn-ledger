# BQN家計簿における取引表現アイデア整理

作成日: 2026-06-15  
目的: BQN/TSV家計簿において、複式簿記・複数ポスティング・封筒予算・日次Cubeをどう扱うか検討するためのアイデア整理。

---

## 0. 前提

現在の家計簿システムでは、正データは TSV とし、人間が直接読める・書けることを重視している。

BQN側は、正データを壊さず、派生ビュー・レポート・検査を作る役割を持つ。

現在の中心的な方向性は以下。

```text
正データ:
  journal.tsv
  plan.tsv
  budget_alloc.tsv
  accounts.tsv
  cycle.tsv

内部表現:
  Day × Account × Layer の Canonical Daily Cube

Layer:
  actual
  plan
  budget
  forecast
```

今回の検討では、特に次の問題を扱う。

- 複数ポスティングをどう扱うか
- 1行1イベントのTSV形式を保てるか
- 複式簿記の借方・貸方をBQN的な配列に溶かせるか
- 封筒予算や食費管理をより自然に扱えるか
- 正データと派生ビューをどう分けるか

---

## 0.1 研究動機

この文書の目的は、単にBQNで家計簿を実装することではない。

複式簿記、ledger形式、複数posting、3式簿記、event log、projection、cube、graphなどでは、会計データをどのような形で表現するべきかについて、さまざまな考え方がある。

この検討では、BQNによる配列処理の立場から、次の作業仮説を置く。

```text
配列として自然に処理できない場合、
処理方法ではなく、前提となるデータ表現がずれている可能性がある。
```

ただし、これはledger形式や複数posting形式を否定するものではない。

ledger形式は、人間の入力、説明、履歴、監査可能性において強みを持つ可能性がある。

一方で、配列処理の観点からは、ledger形式の複数postingを、勘定科目空間上の疎ベクトル表現として再解釈できる可能性がある。

したがって、本検討では、複式簿記の本質をただちに借方・貸方の記法に固定せず、次のような可能性を調べる。

```text
複式簿記の本質は、
各取引が勘定科目空間上の合計0ベクトルであることにあるのではないか。
```

この視点から、取引、予定、予算、封筒、日次レポートを、

```text
Tx × Account × Layer
```

および、

```text
Day × Account × Layer
```

の配列として扱えるかを検討する。

---

## 0.2 オープンエンドな研究姿勢

この文書は、BQN家計簿における取引表現の最終決定ではない。

目的は、ledger形式、複数posting形式、取引ベクトル、daily cube、隣接行列、時系列信号、event log、triple-entry的な証跡モデルなどを比較し、それぞれの表現が何を見やすくし、何を見えにくくするかを観察することである。

ここでは、取引を勘定科目空間上の合計0ベクトルとして捉える方向を、有力な作業仮説として扱う。

ただし、それは他の表現を否定するものではない。

ledger形式は人間の入力や説明に強く、posting集合は複数要素を素直に表現でき、cubeは集計と比較に強く、graphは資金の流れを見やすくし、時系列信号は生活速度や予算消費を観察しやすくする。

したがって、本研究では「どれが正しいか」ではなく、次の問いを中心にする。

```text
どの表現に変換すると、どの問いに答えやすくなるか。
```

この文書は設計決定ではなく、探索地図である。

---

## 0.3 複式簿記を内包するより大きな構造の可能性

この文書では、複式簿記の本質をひとつに固定しない。

複式簿記は、借方・貸方という記法だけで定義されるものではなく、より大きな情報構造の一部として捉えられる可能性がある。

たとえば、会計を次のように見ることができる。

```text
会計とは、出来事を状態変化として記録し、
その変化が特定の不変量を保つかどうかを
検査できるようにする体系である。
```

この視点では、複式簿記は次のように位置づけられる。

```text
複式簿記とは、
金額を対象とする状態変化について、
勘定科目空間上の合計0という不変量を保つように
記録する方法である。
```

この定義では、借方・貸方は本質そのものではなく、合計0の状態変化を人間が扱うための歴史的・実務的な表記形式と見ることができる。

また、ledger形式の複数postingは、勘定科目空間上の疎ベクトル表現として再解釈できる。

3式簿記や監査証跡の考え方は、この状態変化に対して、第三者性、合意、検証可能性を追加する方向として見ることができる。

したがって、本研究では複式簿記を閉じた完成形としてではなく、保存則、状態遷移、証跡、射影、配列変換といった、より大きな構造の中に置いて観察する。

### 本質候補をひとつに固定しない

複式簿記の本質候補として、少なくとも次の見方がある。

```text
候補A:
  借方・貸方という二面性

候補B:
  各取引の合計が0になる保存則

候補C:
  資産・負債・資本・収益・費用の関係を保つ恒等式

候補D:
  取引前後の状態変化を、検査可能な形で残すこと

候補E:
  現実の出来事を、複数の観点に同時に写像すること

候補F:
  組織や生活の活動を、あとから説明可能な構造にすること
```

BQNや配列処理から見ると、候補Bが強く見える。

しかし、人間の生活記録、監査、説明可能性から見ると、候補D、候補E、候補Fも重要である。

したがって、この文書では複式簿記を「合計0ベクトル」だけに還元しない。

「合計0ベクトル」は、有力な観察レンズのひとつとして扱う。

---

## 0.4 この仮説が閉じすぎないために

この文書では、取引を勘定科目ベクトルとして扱う方向を強く検討する。

しかし、この方向はあくまでBQNによる配列処理から見た作業仮説であり、会計データ表現の唯一の正解ではない。

ledger形式、複数posting形式、journal形式、event log形式、triple-entry的な証跡モデル、graphモデル、時系列信号モデルには、それぞれ異なる目的に対する強みがある。

したがって、この文書では次の姿勢を取る。

- ベクトル化は有力仮説として扱う
- ledger形式は疎ベクトル表記として再解釈できるか検討する
- ただし、人間の入力形式としてのledgerの強みは否定しない
- 配列処理しにくい形式を即座に誤りとは見なさない
- その形式が何に最適化されているのかを先に観察する
- 実装上の美しさと生活記録としての自然さを分けて評価する

## 0.5 遠い数学的アナロジーと将来仮説

この節では、今すぐの実装方針ではなく、将来役に立つかもしれない数学的な観察レンズを保管する。

ここに挙げる概念は、現時点では採用決定ではない。

また、複式簿記や家計簿をこれらの理論へただちに還元できると考えるものでもない。

目的は、会計データ表現を別の数学的構造から眺めたときに、どのような問いが立つかを残しておくことである。

### 線形代数とゼロサム制約

現在の検討に最も近いのは、線形代数的な見方である。

```text
1取引 = 勘定科目空間上のベクトル
複式簿記の貸借一致 = 取引ベクトルの成分和が0
残高 = 取引ベクトルの累積和
```

この見方は、BQNの配列処理と相性が良い。

ledger形式の複数postingは、このベクトルのうち非ゼロ成分だけを列挙した疎表現として扱える可能性がある。

### キルヒホッフ的な保存則

お金の流れを回路網のように見ると、入出金の保存則をキルヒホッフの法則に似た形で考えられる可能性がある。

```text
ある節点に入る量と出る量の関係を見る
閉じた系では総量が保存される
外部からの収入や外部への支出は境界として扱う
```

これは、複式簿記を保存則を持つ状態遷移ログとして見る考えと近い。

ただし、家計では収入、消費、予算配分、資産移動が混ざるため、どの範囲を閉じた系と見るかを先に定義する必要がある。

### グラフラプラシアン

口座や封筒をノード、資金移動をエッジとして見ると、資金フローをグラフとして扱える。

その派生として、グラフラプラシアンを使う可能性がある。

```text
ノード:
  accounts / envelopes

エッジ:
  資金移動 / 消費 / 配分

見るかもしれないもの:
  連結性
  流れの偏り
  循環
  特定ノードへの集中
```

ただし、固有値やラプラシアンで家計のボトルネックや流動性がすぐに分かるとは限らない。

これは本体の集計エンジンではなく、cashflow graph から派生する観察ビューとして扱うのが安全である。

### ホモロジー代数、チェイン、境界

ホモロジー代数の `境界の境界はゼロ` という考え方は、会計の保存則や閉じた精算サイクルを考える比喩として興味深い。

```text
∂ ∘ ∂ = 0
```

ただし、これをそのまま複式簿記の貸借一致と同一視してはいけない。

複式簿記の基本的な一致は、まずは次の形で表せる。

```text
各取引ベクトルの成分和 = 0
```

ホモロジー的に扱うには、少なくとも次の対応を定義する必要がある。

```text
何を0-chainと見るか
何を1-chainと見るか
何を境界作用素と見るか
精算サイクルや立替関係をどのチェインとして表すか
```

そのため、現時点では遠い将来仮説として扱う。

### ベッチ数と循環の監視

家計内の立替、精算、封筒間移動をグラフとして見た場合、意図しない循環が生じることがあるかもしれない。

このとき、第1ベッチ数のようなトポロジー的不変量を、循環の数を眺める比喩として使える可能性がある。

ただし、通常の家計管理では、循環そのものが問題とは限らない。

精算ルート、立替関係、クレジットカード利用と支払日など、周期や遅延を含む構造を扱う場面で、将来的に検討する余地がある。

### 圏論と関手

Datalogのfact、BQNのarray、ledger形式、TSV形式は、それぞれ異なる表現である。

同じ正データから複数の表現を生成する場合、その変換規則を構造を保つ写像として見られる可能性がある。

```text
TSV / journal
  -> Datalog facts
  -> BQN arrays
  -> reports
```

これを圏論の言葉でいえば、関手のようなものとして見られるかもしれない。

ただし、現時点では「関手を実装する」と考えるより、次のように扱う方が安全である。

```text
同じ正データから、
Datalog向けfactとBQN向けarrayを生成する変換規則を整理する。
```

その変換が何を保存し、何を捨て、何を見やすくするのかを観察する。

### 力学系と信号処理

この中で比較的実用に近いのは、力学系や信号処理の見方である。

日々の支出や残高を時系列として扱えば、生活速度や予算消費を観察できる。

```text
日次支出 = 信号
累積支出 = Scan
支出速度 = 差分または移動平均
残り予算 ÷ 残日数 = 安全速度
```

これは食費管理、封筒予算、固定費予測に接続しやすい。

### 熱力学やエントロピーの比喩

固定費の支払い、日々の生活費、予算の消費を、エネルギーの散逸や分布の広がりとして見る比喩もありうる。

ただし、これは現時点では実装や検査に直接つながるものではない。

使う場合は、次のような問いに限定する。

```text
支出が少数カテゴリに集中しているか
支出が広く散らばっているか
予算配分がどの程度ばらけているか
```

これはエントロピーという語を使わなくても、カテゴリ分布や集中度として扱える可能性がある。

### この節の扱い

この節の概念は、次の3段階に分けて扱う。

```text
実用候補:
  N次元ベクトル
  ゼロサム制約
  daily cube
  信号処理 / burn rate

派生ビュー候補:
  graph
  adjacency matrix
  graph Laplacian
  Kirchhoff的保存則

遠い研究仮説:
  homology
  Betti number
  category theory
  functor
  entropy
```

重要なのは、これらを「究極の理論」として扱わないことである。

これらは、会計データを別の角度から観察するための、将来用の棚である。

## 1. トランザクションのN次元ベクトル化

### 概要

すべての勘定科目を要素とする巨大な1次元配列、つまりベクトルを作る。

1回の取引は、各勘定科目に対する増減を持つベクトルとして表現する。

このベクトルは、合計が必ず0になる。

```text
1取引 = 勘定科目ごとの増減ベクトル
各取引行の合計 = 0
全体残高 = 取引ベクトルを足し合わせるだけ
```

複式簿記の借方・貸方を明示的に持つのではなく、符号付きの配列として表す。

### 例

```tsv
date	tx_id	desc	actual:bank	actual:food	actual:income	budget:daily	budget:reserve
2026-06-15	t001	食材	-800	800	0	-800	0
2026-06-15	t002	年金	100000	0	-100000	0	0
2026-06-15	t003	予算配分	0	0	0	30000	70000
```

### BQNとの相性

非常に良い。

BQNでは、取引ごとのベクトルを積み重ねることで、

```text
Tx × Account × Layer
```

のような配列として扱える。

そこから日次に畳み込めば、

```text
Day × Account × Layer
```

の Canonical Daily Cube へ自然につながる。

### 強み

- 複数ポスティングを自然に表現できる
- BQNの配列処理と相性が良い
- 合計0チェックが単純になる
- 残高計算がFoldやScanで表せる
- 借方・貸方を符号に変換できる
- 正規化された内部表現にしやすい

### 弱み

- 勘定科目が増えるとTSVが横に広がる
- 人間が手書きするには少し重い
- 空欄や0が大量に出る
- 表としては整っているが、生活記録としては少し機械寄りになる

### 位置づけ

有力候補。

ただし、正データそのものにするか、内部表現にするかは慎重に検討する。

おすすめは、まず内部表現または実験ブランチで試すこと。

## 2. 隣接行列・テンソルによる流体モデリング

### 概要

勘定科目をノードとし、取引をノード間の有向エッジとして扱う。

つまり、お金の動きをグラフとして見る。

```text
勘定科目 = ノード
取引 = ノード間の有向エッジ
金額 = エッジの重み
```

日付も軸に加えると、

```text
Day × From × To
```

または、

```text
Day × SourceAccount × DestinationAccount
```

の3次元テンソルとして扱える。

### 例

```text
bank    -> food      800
income  -> bank   100000
reserve -> daily   30000
```

### BQNとの相性

派生ビューとしては良い。

期間ごとの資金移動、食費への流入、封筒間の移動などを行列・テンソルとして集計できる。

### 強み

- お金の流れが視覚的にわかりやすい
- cashflowレポートに向いている
- 封筒間移動を表現しやすい
- グラフや水路図のような分析に向いている
- From/Toが明確な取引では強い

### 弱み

- 複式簿記の全取引が自然なFrom/Toになるとは限らない
- 収入・費用・資産・負債・予算が混ざると意味が曖昧になる
- 正データにすると、会計的な厳密さが崩れる可能性がある
- N対N取引では結局分解ルールが必要になる

### 位置づけ

正データにはしない方がよい。

ただし、派生ビューとしては有力。

おすすめは、

```text
journal / vector / cube
  -> cashflow graph
  -> adjacency matrix
```

の順で生成すること。

## 3. 1対1原則と仮想精算口座

### 概要

複数ポスティングやN対N取引をすべて1対1の取引に分解する。

複雑な取引は、一時的に存在する仮想的な中継口座、つまりClearing Accountを経由させる。

```text
複雑な取引
  -> clearing account を経由
  -> 1対1取引の集合へ分解
```

### 例

給与が入金され、同時に天引きや予算配分がある場合。

```text
income -> clearing
clearing -> bank
clearing -> tax
clearing -> reserve
clearing -> daily
```

### BQNとの相性

内部処理としてはかなり良い。

すべての取引が1対1になるため、配列・行列・グラフ化が簡単になる。

### 強み

- 処理が単純になる
- From/To形式に落とし込みやすい
- 隣接行列化しやすい
- 複雑な取引を機械的に分解できる
- 中間表現として扱いやすい

### 弱み

- 人間が読む正データとしては冗長になる
- 1つの生活イベントが複数行に分かれて見えにくくなる
- clearing account が増えると意味が散らかる
- 正データにすると、生活記録としてのまとまりが弱くなる

### 位置づけ

正データではなく、内部変換用の中間表現として有効。

おすすめは、

```text
人間用の正データ:
  tx_id で同じイベントをまとめる

機械用の派生データ:
  clearing account を使って 1対1 に正規化する
```

という分離。

## 4. 家計の信号処理パラダイム

### 概要

取引を点として見るだけでなく、日々のお金の動きを時系列信号として扱う。

支出速度、累積支出、残り予算、残日数などを、波形や速度として見る。

```text
日次支出 = 信号
累積支出 = 積分 / Scan
日ごとの差分 = 微分 / Difference
残り予算 ÷ 残日数 = 安全速度
```

### 例

食費管理では、単に「残りいくら」ではなく、

```text
現在の消費速度
安全な消費速度
予算超過までの距離
```

を見る。

```text
実消費速度 > 安全速度
  -> 警戒

実消費速度 <= 安全速度
  -> 安全圏
```

### BQNとの相性

非常に良い。

BQNのScan、差分、集計、窓関数的な処理と相性が良い。

### 強み

- 食費管理に向いている
- 封筒予算の燃焼速度がわかる
- 固定費予測にも使える
- 日次Cubeから自然に作れる
- 生活のリズムやブレを観察できる
- レポートとして実用性が高い

### 弱み

- 正データの形式ではない
- 会計記録そのものではなく分析手法
- 計算結果の意味づけが強くなりすぎる可能性がある
- AIが勝手に解釈を盛りがちな領域でもある

### 位置づけ

正データではなく、レポート思想として非常に有力。

特に食費・日次予算・生活速度の観察に向いている。

おすすめは、

```text
journal
  -> daily cube
  -> daily signal
  -> burn rate report
```

の流れ。

## 5. 資金パーティクル粒子追跡モデル

### 概要

金額を単なる数値ではなく、1円単位の粒子として考える。

取引とは、粒子がどの口座・封筒・場所にいるかを書き換える操作と見る。

```text
1円 = 粒子
口座・封筒 = 座標
取引 = 粒子の座標移動
残高 = 各場所にいる粒子数
```

### 例

```text
1000個の粒子が bank から daily に移動
800個の粒子が daily から food に移動
```

### BQNとの相性

概念遊びとしては面白い。

しかし、実装としては過剰になりやすい。

5万円なら5万粒子、10万円なら10万粒子を扱うことになる。

### 強み

- 封筒予算の直観に合う
- お金の所在を物理的に感じやすい
- 粒子群として考えると面白い
- GroupやIndicesの発想とつながる
- 教材・比喩・実験には向いている

### 弱み

- 実装としては重い
- 実用的な家計簿には過剰
- 金額を整数で持つだけで十分な場面が多い
- 正データにすると複雑すぎる
- 本番システムに入れると見通しが悪くなる

### 位置づけ

本番には入れない方がよい。

ただし、思想・教材・遊び・比喩としては有用。

もし実装するなら、1円単位ではなく、

```text
amount + location + tag
```

を持つロット管理のようにするのが現実的。

## 6. 追加案: スパース・ベクトルTSV

### 概要

横長の全勘定列を毎回持つ代わりに、1行1イベントの中に、変更のあった勘定だけを記録する。

```tsv
date	tx_id	desc	vector
2026-06-15	t001	食材	actual:bank=-800;actual:food=800;budget:daily=-800
2026-06-15	t002	年金	actual:bank=100000;actual:income=-100000
```

### 強み

- 1行1イベントを保てる
- 複数ポスティングを表現できる
- 勘定科目が増えても列が爆発しない
- TSVとして読める
- BQN側で密なベクトルに展開できる

### 弱み

- `vector` 欄の中に小さなDSLが生まれる
- 素朴なTSVではなくなる
- パーサが必要になる
- 手書きミスの検査が重要になる
- BQNだけで扱うには少し面倒かもしれない

### 位置づけ

1行1イベントを重視するなら有力。

ただし、現在のTSVの素朴さを守るなら、すぐに採用せず実験扱いがよい。

## 8. 推奨アーキテクチャ

現時点では、正データを急に大改造しない方がよい。

おすすめは、次のように層を分けること。

```text
正データ:
  journal.tsv
  plan.tsv
  budget_alloc.tsv

正規化:
  Tx × Account × Layer vector

日次集計:
  Day × Account × Layer cube

派生ビュー:
  balance report
  cashflow graph
  envelope report
  burn rate report
  food safety speed report
```

### 流れ

```text
journal / plan / budget_alloc
  -> transaction vector
  -> daily cube
  -> reports
```

この形にすると、

- 複式簿記の整合性
- BQNの配列処理
- 人間が読むTSV
- 封筒予算
- 食費の燃焼速度
- cashflow可視化

を同時に扱いやすくなる。

ただし、これは現時点の有力な作業経路であり、最終的な唯一解ではない。

## 9. 正データ候補ごとの整理

date	tx_id	desc	layer	account	amount

## 10. 現時点の整理

現時点で有力なのは、N次元ベクトル化である。

ただし、それをすぐ正データにする必要はない。

まずは、

```text
既存TSV
  -> Tx × Account × Layer vector
  -> Day × Account × Layer cube
```

という内部変換として試すのがよい。

隣接行列・信号処理・粒子モデルは、正データではなく派生ビュー・分析・教材として扱う。

特に食費管理には、信号処理パラダイムが強い。

```text
残り予算
残り日数
現在の消費速度
安全な消費速度
```

を出せば、封筒予算よりも実感に近いレポートになる可能性がある。

ただし、この整理は結論ではなく、現時点での観察である。

今後、ledger形式、event log形式、triple-entry的な証跡モデル、あるいは別の表現の方が、特定の問いに対してより自然に働く可能性も残す。

## 12. 最終メモ

今回の整理では、借方・貸方を捨てるというより、借方・貸方を符号付き配列へ変換する方向を有力仮説として扱う。

複式簿記の骨格は保つ。

ただし、表現はBQNに合わせて変える。

```text
簿記の整合性:
  合計0

BQNの美しさ:
  配列として足せる

人間の読みやすさ:
  TSVとして確認できる

生活管理:
  食費や封筒の速度を見る
```

この4つを同時に満たすには、

```text
正データは人間寄り
内部表現は配列寄り
レポートは生活寄り
```

に分けるのが良い。

ただし、これは閉じた結論ではない。

複式簿記は、より大きな情報構造の中に内包される可能性がある。

その大きな構造は、保存則、状態遷移、証跡、射影、配列変換、生活イベントログを含むかもしれない。

この文書は、正しい形式をひとつ決めるためのものではなく、会計データをさまざまな表現へ変換し、それぞれの表現が何を見せ、何を隠すのかを観察するための探索地図である。
