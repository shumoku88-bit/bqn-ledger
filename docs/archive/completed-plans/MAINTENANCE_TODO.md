## Phase 1: BQN単体テスト基盤の構築（課題1）
**目的**: AIがBQNのコードを修正した際に、配列の次元や挙動が壊れていないかを局所的に自動検証できるようにする。全体スナップショットテストの前に失敗を検知する足場を作る。

- [x] **1-1. テストフレームワーク（極小）の作成**
  - `tools/test_lib.bqn` を新規作成する。
  - シンプルなアサーション関数 `Assert`、`AssertEq`（期待値と実態の比較、失敗時に呼び出し元と値を表示して `•State 1` で終了）を実装する。
- [x] **1-2. `core.bqn` の単体テスト作成**
  - `tests/test_core.bqn` を新規作成する。
  - `SplitKeepEmpty`、`ToNum`、`GetTxUpd` など、すでに実績のある純粋関数の入出力を検証するテストを書く。
- [x] **1-3. テストランナーの統合**
  - `./tools/check.sh` の先頭（スナップショットテストの前）に、`tests/test_*.bqn` をすべて実行するステップを追加する。
- **完了条件**: `./tools/check.sh` を実行した際、単体テストが走り、意図的にテストを失敗させるとスクリプトが落ちること。
- **次のAIへの引継ぎメモ**: 「Phase 1完了。BQNの単体テストが `tests/` ディレクトリで動くようになりました。以降の修正では、必要に応じて単体テストを追加・実行してから `check.sh` を回してください。」

## Phase 2: メタデータLinter（スキーマ検証）の構築（課題3）
**目的**: 6列目以降の `key=value` のタイポや不正な値を実行前（または入力時）に弾き、サイレントエラーを防ぐ。

- [x] **2-1. メタデータ定義（スキーマ）ファイルの作成**
  - `config/meta_schema.tsv`（または `docs/JOURNAL_META.md` をパース可能な形式に整理）を作成する。
  - 定義例：キー名（`tax`）、許容値（`business|private|mixed` または `enum`）、型（`number` / `string`）。
- [x] **2-2. Linterロジックの実装**
  - `tools/lint_meta.bqn` を新規作成、または `core.CheckRow` を拡張する。
  - 6列目以降を `key=value` でパースし、スキーマ定義に存在しないキーや、不正な値が含まれていないかチェックする関数を作る。
- [x] **2-3. 既存データのチェックとクリーンアップ**
  - 現在の `journal.tsv` や `plan.tsv` に対して Linter を実行（まずはDry-runや警告出力のみ）する。
  - もし過去のデータにタイポ（例：`biz=ok` なのに数字が期待されている等）があれば、ユーザーに確認して修正する。
- [x] **2-4. CI / 入力ツールへの統合**
  - Linter を `./tools/check.sh` に組み込む。
  - （オプション）`tools/add.bqn`（入力補助ツール）のバリデーションにも組み込み、入力時にエラーを出すようにする。
- **完了条件**: 未定義のメタデータキー（例：`txa=private`）を `journal.tsv` に追記した際、`check.sh` がエラーを吐いて停止すること。
- **次のAIへの引継ぎメモ**: 「Phase 2完了。メタデータの厳密なチェック機構が入りました。今後新しいメタデータキーを追加する場合は、必ず `config/meta_schema.tsv` と `docs/JOURNAL_META.md` を更新してください。」

## Phase 3: ドメインルール（生活設定）の分離（課題4）
**目的**: 「給料日」「基準サイクル」「特別な月」などの生活密着ルールをコード内から追い出し、設定ファイル化する。将来、利用者の状況が変わってもBQNコードを触らずに済むようにする。

- [x] **3-1. ハードコードされた設定値の洗い出し**
  - `cycle.bqn`、`engine/report_cashflow_due.bqn`、`main.bqn` 等から、「15日始まり」や「特定の曜日・祝日処理」など、生活由来のマジックナンバーやルールをリストアップする。
- [x] **3-2. `config.tsv`（全体設定ファイル）の作成**
  - `config.tsv` をルート（または `config/`）に作成する。
  - 例: `CYCLE_START_DAY=15`、`DEFAULT_PAYMENT_DUE=25` など。
- [x] **3-3. 設定読み込みモジュールの実装**
  - `core.bqn` または `config.bqn` に、`config.tsv` を読み込んでグローバルな連想配列（または名前空間）として提供する関数を作る。
- [x] **3-4. ロジックの設定ファイル参照への置き換え**
  - 洗い出したマジックナンバーを、読み込んだ設定値に置き換える。
  - 置き換え後、必ず単体テストおよび `./tools/snapshot.sh` を実行し、既存の出力（回帰）に一切変化がないことを確認する。
- **完了条件**: `config.tsv` の `CYCLE_START_DAY` を変更すると、レポートの出力サイクルが連動して変わること。かつ、元に戻せばスナップショットテストが完全にパスすること。
- **次のAIへの引継ぎメモ**: 「Phase 3完了。生活ルールが `config.tsv` に分離されました。今後生活環境の変化（給料日の変更など）があればコードではなく設定ファイルを変更してください。」
