# BQN家計簿 AIフレンドリー化リファクタリング計画 (TODOリスト)

**目的**: BQN特有の「AIの苦手分野（配列の次元の暗黙性、密度の高い巨大ファイル、エラー時のパニック）」を克服し、AI（pit）と人間が完全に安心して拡張できる最終形態へ引き上げる。

## Phase 1: 配列の次元・形状 (Shape/Rank) のドキュメント化（課題1）
**課題**: BQNには型がないため、AIが「この変数には1次元リストが来るのか、2次元行列が来るのか」を誤解してコードを壊しやすい。
**対策**: 主要関数の直前に、入出力の形（Shape）をコメントで明記する。

- [x] **1-1. `core.bqn` のShape明記**
  - `GetTxUpd`、`SplitKeepEmpty` などの主要関数の引数 `𝕩` と戻り値の形を `[N × 5] strings` のようにコメントで追記する。
- [x] **1-2. データロード層 (`report_tx_updates.bqn`) のShape明記**
  - `BuildCube` が返す `cube_balances` や `cube_updates` が `[Day × 256 × 4] numbers` であることなどを明記する。
- [x] **1-3. エンジン層 (`report_engine.bqn`) の戻り値スキーマの明記**
  - `main.bqn` に渡される巨大な `record` の中身（どのキーにどんなShapeのデータが入っているか）をファイル先頭に定義として書く。
- **完了条件**: 主要なBQNファイルを開いたとき、関数の役割だけでなく「データの形」が一目でわかるようになっていること。

## Phase 2: 巨大ファイル (`report_sections.bqn`) の分割（課題2）
**課題**: 23KBを超える `report_sections.bqn` はAIの認知負荷が高すぎ、一部を直すだけで全体を壊すリスクがある。
**対策**: セクションごとにファイルを細かく分割する。

- [x] **2-1. `report_sections/` ディレクトリの作成**
  - 機能拡張を想定し、表示ロジックを格納する専用フォルダを作る。
- [x] **2-2. 各セクションの切り出し（モジュール化）**
  - `envelopes`, `outlook`, `daily-trend`, `cashflow`, `cycle-consult` などの巨大な関数を、それぞれ `report_sections/sec_envelopes.bqn` のように独立したファイルに切り出す。
- [x] **2-3. ルーター (`report_sections.bqn`) の再構築**
  - `report_sections.bqn` は実ロジックを持たず、分割した各ファイルを取り込んで `main.bqn` に提供するだけの薄いルーター（エントリーポイント）にする。
- **完了条件**: 分割後も `./tools/check.sh` が完全にパスし、`main.bqn --section all` の出力が1文字も変わらないこと。

## Phase 3: AI用サンドボックスと開発プロセスの標準化（課題3）
**課題**: AIがいきなり本番の集計ロジックを修正しようとすると、BQNのエラーログにパニックを起こしてドツボにハマる。
**対策**: 「実験室」を用意し、AIが必ずそこで試作品を作ってから本番に適用するルールを設ける。

- [ ] **3-1. サンドボックスの雛形 (`tests/sandbox_template.bqn`) の作成**
  - 最小限のモックデータ（数行の配列）と、実行結果を表示するだけの「安全な実験用スクリプト」の雛形を作る。
- [ ] **3-2. `AGENTS.md` への「サンドボックス駆動開発」ルールの追記**
  - 「新しい集計ロジックや複雑な配列操作を書くときは、いきなり本番ファイルを直さず、まず `sandbox.bqn` を作ってモックデータで実験せよ」
  - 「サンドボックスでの実行結果が正しく通ってから、本番ファイルに適用せよ」というルールを明記する。
- **完了条件**: AIが `AGENTS.md` を読んだとき、「複雑な実装は必ずサンドボックスで実験してから本番に反映する」という手順を強制されること。
