# Decision: Multi-Posting Support Policy

作成日: 2026-06-21
決定日: 2026-06-22
ステータス: 方針決定 / 本体は A-1 採用

---

## 0. 最終決定（2026-06-22）

`bqn-ledger` 本体では、複数ポスティング対応を **A-1: `txn_id` メタデータによる複数行束ね** として進める。

決定内容:

1. **本体では A-1 を採用する。**
   - 既存の `date / memo / from / to / amount` という先頭5列契約を維持する。
   - 1行は従来どおり「1対1の資金移動」として扱う。
   - 同じ現実イベント、同じレシート、同じ銀行明細に属する複数行は、6列目以降の `txn_id=...` で束ねる。
   - `BuildCube` の意味は変更しない。

2. **A-2 は必要になった場合に拡張できる余地として残す。**
   - `virtual:suspense` のような中継・諸口アカウントを使う方式は、標準方式にはしない。
   - ただし、銀行明細1件と内訳を強く対応させたい場合や、中継口座のゼロ残高検証が必要になった場合に使える互換パターンとして残す。
   - A-2 を使う場合も、先頭5列契約と `txn_id` 束ねは維持できる。

3. **A-3 は本体に入れず、フォークまたは別リポジトリで研究する。**
   - A-3 は実質的に「1行1ポスティング」へ近づくため、`from / to / amount` の意味を大きく変える。
   - `BuildCube` 前段、パース処理、ゼロサム検証、Event IR / Projection IR の見直しが必要になる。
   - 本体の安定性を守るため、A-3 は `bqn-ledger` 本体ではなく、別プロジェクトとして進める。

この決定により、本体は「生活で使う安定した正本レポートエンジン」として守り、A-3 は「posting 構造を研究する別系統」として自由に掘る。

---

## 1. 課題：複数ポスティングの必要性

現在の `bqn-ledger` の取引データ形式（`journal.tsv`, `plan.tsv`）は、1行で「`from` 勘定から `to` 勘定へ `amount` を移す」という1対1の移動を前提としている。

しかし、実際の家計簿運用では、3つ以上の勘定科目が同時に動く取引（複数ポスティング / マルチポスティング）が発生する。
* **例1（支払いの混在）**: スーパーで食費800円と日用品300円を同時に買い、SMBC口座から1100円支払った。
* **例2（給料と控除）**: 額面20万円の給料に対して、税金3万円が引かれ、17万円が銀行口座に振り込まれた。

現在、これらの取引を記録するには「手動で複数の1対1取引に分割する」か「一時的な仮勘定を経由させる」必要があるが、より直接的かつ安全に複数ポスティングを扱えるようにしたい。

---

## 2. 検討アプローチ

以下の3つのアプローチを検討した。

### アプローチA. メタデータ `txn_id` による複数行の束ね（本体採用）
既存の「1行1取引（TSVスキーマ）」を維持したまま、メタデータ列にトランザクションID（`txn_id`）を付与し、同一取引を複数行に分けて記述する。
```tsv
2026-06-01  スーパー  assets:smbc  expenses:食費    800  txn_id=2026-0001
2026-06-01  スーパー  assets:smbc  expenses:日用品  300  txn_id=2026-0001
```
* **メリット**:
  * 既存の BQN 計算エンジン（`BuildCube`）は一切変更なしでそのまま動く。
  * 既存の「1行1取引」データと完全に混在・併存できる。
  * Go editor やUI側で「同じ `txn_id` を持つ行」を自動生成・一括表示することが比較的容易。
* **デメリット**: 記述において `from` 口座や日付などが重複し、データがやや冗長になる。

### アプローチB. 仲介仮勘定（諸口・サスペンス口座）の運用（互換拡張として保留）
一時的な中継口座（例: `virtual:suspense`）を導入し、1対1の仕訳だけで中継して記録する。
```tsv
2026-06-01  スーパー  assets:smbc       virtual:suspense  1100
2026-06-01  スーパー  virtual:suspense  expenses:食費      800
2026-06-01  スーパー  virtual:suspense  expenses:日用品    300
```
* **メリット**: システムの変更やTSVスキーマの拡張が完全に不要。
* **デメリット**: 取引ごとに不要な中継行が大量に発生し、仕訳データが見づらくなる。

### アプローチC. 複数行ポスティング型TSVへのデータ移行（本体では不採用、別repoで研究）
1つの取引を複数行で表し、各行で「どのアカウントがいくら増減したか」を記述する Beancount / Ledger に近いデータ構造へ完全移行する。
```tsv
date        tx_id   desc    account        amount
2026-06-15  t001    食材    assets:smbc    -1100
2026-06-15  t001    食材    expenses:食費   800
2026-06-15  t001    食材    expenses:日用   300
```
* **メリット**: 取引の表現として最も美しく、冗長性がない。
* **デメリット**: パーサ、計算コア、検証ツールなどをすべて大改修する必要がある。

## 3. 計算エンジン（BQN）の互換性と影響範囲

「アプローチC」を採用した場合でも、**エンジンを1から別プロジェクトとして作り直す必要はない。**

### 日次残高Cubeの再利用
表示ロジックや各 View（`liquid_view.bqn`, `envelope_view.bqn` など）が受け取るのは、最終的に計算された `Day × Account × Layer` の日次残高Cubeである。インプットが複数ポスティングになっても、日付ごとに各口座の `amount` の累計を求めれば Daily Cube は問題なく再構築できる。したがって、**レポート出力側のコードは一切書き直す必要がない**。

### 改修が必要な箇所
* **入力・パース処理**: `from / to / amount` から `account / amount(符号付き)` へのパーサー変更。
* **取引ベクトル中間層の明示化**: `CUBE_EVOLUTION_POLICY.md` でも構想されていた、Event IR から Daily Cube に畳み込む前段としての `Tx × Account × Layer` ベクトル（取引単位 of 変化）の導入。
* **ゼロサム（貸借一致）の検証**: `txn_id` ごとに、全行の `amount` 合計が `0` になっているかを検査するバリデータの追加。

この影響範囲が大きいため、アプローチC / A-3 は本体では扱わず、フォークまたは別リポジトリで研究する。

## 4. 本体での段階的な進め方

本体では A-1 を採用する。大改造によるバグの混入を防ぐため、以下の順序で段階的に進める。

1. **ステップ1：`txn_id` 運用契約の固定**
   * 実データは1対1取引のままで、必要な複数行に `txn_id` メタデータを付与する。
   * 同じ `txn_id` 内で揃えるべき値（date / memo / party / from など）を決める。
   * 1行だけの `txn_id` を許容するか、警告するか決める。

2. **ステップ2：A-1 用 lint / helper の強化**
   * BQN側では `txn_id` グループを読んで、日付・memo・party・from の揺れを警告できるようにする。
   * A-1 では各行が単体でゼロサムなので、グループ全体の金額0検証よりも、同一束としての整合性検査を重視する。
   * 必要に応じて、レシート総額や銀行明細との照合に使える summary を追加する。

3. **ステップ3：Go editor 側の入力サポート**
   * Go editor 側で、同じ `txn_id` の複数行を一括入力・表示できる支援UIを検討する。
   * ただし、Go は accounting engine にならず、あくまで source TSV editor として扱う。

4. **ステップ4：A-2 の必要性を運用で判断**
   * 銀行明細1件と内訳を厳密に対応させたい場面が増えた場合、`virtual:suspense` のような中継アカウントを追加するか検討する。
   * A-2 は標準方式ではなく、A-1 と互換な拡張パターンとして扱う。

## 5. txn_id lint 設計に向けた詳細検討（2026-06-22 追記）

複数ポスティングを `txn_id` で束ねるにあたり、BQN側の lint（検証）でどのような「ゼロサム（貸借一致）」を定義するか、以下の3パターンを検討した。

### パターンA-1: 1対1移動のまま `txn_id` で束ねる（本体採用）
```tsv
2026-06-01  スーパー  assets:smbc  expenses:食費    800  txn_id=2026-0001
2026-06-01  スーパー  assets:smbc  expenses:日用品  300  txn_id=2026-0001
```
* **ゼロサム判定**: 各行が単体で `from` (-800) ＋ `to` (+800) ＝ 0 と完結しているため、グループ合計も常に 0。
* **検証項目 (lint)**: 同一 `txn_id` グループ内での日付・説明の完全一致や、引き落とし元（`from`）がすべて同一口座であることなどのビジネスルール検証が主。
* **メリット**: BQNエンジン（`BuildCube`）の改修が不要。
* **デメリット**: 実際の銀行明細（1100円の1件の引き落とし）と金額を直接一致させるのが難しくなる（行が分かれるため）。

### パターンA-2: 諸口中継口座（`virtual:suspense`）を利用する（互換拡張として保留）
```tsv
2026-06-01  スーパー  assets:smbc       virtual:suspense  1100  txn_id=2026-0001
2026-06-01  スーパー  virtual:suspense  expenses:食費      800   txn_id=2026-0001
2026-06-01  スーパー  virtual:suspense  expenses:日用品    300   txn_id=2026-0001
```
* **ゼロサム判定**: 同一 `txn_id` グループ内で、中継口座 `virtual:suspense` に対する増減の総和が 0 になっていることを検証。
  * 行1: `virtual:suspense` ＝ +1100（`to` に指定）
  * 行2: `virtual:suspense` ＝ -800（`from` に指定）
  * 行3: `virtual:suspense` ＝ -300（`from` に指定）
  * 合計 ＝ 1100 - 800 - 300 ＝ 0
* **検証項目 (lint)**: 中継用のアカウント（例: `virtual:*`）が、取引の最後で必ず残高 0 になることを静的に担保。同一グループ内での日付・説明の一致。
* **メリット**: 既存の `BuildCube` は一切変更なしでそのまま動き、他のビューへの悪影響がない。
* **デメリット**: 記述が3行になり冗長。

### パターンA-3: 片側仕訳（FromかToのいずれかを空）の許容（本体では不採用）
```tsv
2026-06-01  スーパー  assets:smbc          1100  txn_id=2026-0001
2026-06-01  スーパー               expenses:食費  800   txn_id=2026-0001
2026-06-01  スーパー               expenses:日用  300   txn_id=2026-0001
```
* **ゼロサム判定**: 同一 `txn_id` 内の全行の金額（`from` に書かれた場合はマイナス、`to` に書かれた場合はプラス）の総和が 0 になっていることを検証。
* **検証項目 (lint)**: 同一 `txn_id` グループ内の日付・説明の一致、および金額総和が 0 であること。
* **メリット**: 取引の表現がすっきりし、アプローチC（1行1口座）に非常に近くなる。
* **デメリット**: `BuildCube` 側で「`to` または `from` が空である行」を正しくパースして残高に畳み込めるように改修が必要。

A-3 は本体の先頭5列契約を変えるため、このリポジトリでは採用しない。必要ならフォークまたは別リポジトリで、posting-based engine として研究する。
