# サイクル相談レポート機能案

この文書は、2026-06-06 の家計相談を分解し、BQN 家計簿にどんなレポート機能があると相談しやすいかを整理するための設計メモです。

当初は「生活防衛資金を 100,000円に戻す」ことを主目的として考えたが、相談を分解すると、より手前の目的は次の通りでした。

```text
次サイクルを、貯金の取り崩しなしで終える。
そのために、収入日の配分・前サイクルの振り返り・次サイクルの見込みを整理する。
```

したがって、この文書では「貯金10万円達成」を主役にせず、まず **サイクル相談** を主役にします。


## 必要な機能

## A. 収入日配分ビュー

収入日に「まず何を確保するか」を見るビュー。

表示例:

```text
【収入日配分】
次回収入日: 2026-06-15
収入予定  : 231,916

[収入直後に確保するもの]
家賃              64,000
光熱費(2ヶ月見込) 30,000
借金返済          10,000
gpt-plus           3,000
合計             107,000

[その後の予定]
wifi               4,812
povo                 330
健康保険料         2,000
google-one         1,450
合計               8,592

予定支出合計     115,592
生活費に残る額   116,324
```

ポイント:

- `plan.tsv` に明示された予定だけを見る
- 自動で未来行を生成しない
- 推測より、現在の予定表の妥当性を確認する

---

## B. 前サイクル振り返りビュー

既存の `cycle` セクションに近いが、「相談用」に要点をまとめる。

表示例:

```text
【前サイクル振り返り】
期間: 2026-04-15 〜 <2026-06-15
収入: 232,480
支出: 221,441
収支:  11,039

[大きい支出]
家賃          64,000
光熱費        28,500
食費          29,869
タバコ        23,808
通信          14,456
AIサブスク    11,687
予備          10,318
```

既存の `cycle` を大きく変えるより、必要なら `cycle-consult` のような別セクションで出す。

## C. 貯金/取り崩しフロー

「黒字だったか」だけではなく、サイクル中に貯金を入れたり戻したりした流れを見たい。

表示例:

```text
【貯金/取り崩しフロー】
assets:ゆうちょ
  入金       : +10,000
  取り崩し   : -10,000
  純増減     :       0

assets:オルカン積立
  入金       :  +3,000
  取り崩し   :       0
  純増減     :  +3,000
```

このビューの問い:

```text
前サイクルは、貯めたお金が実際に残ったか？
```

---

## D. 次サイクル予定ビュー（登録済みベース）

`plan.tsv` に書かれた予定だけから、次サイクルの予定収入・予定支出・差引を出す。
これは「サイクル終了時見込み」ではなく、登録済み予定だけの事実セットとして扱う。

表示例:

```text
【次サイクル予定（登録済みベース）】
期間: 2026-06-15 〜 <2026-08-14
予定収入: 231,916
予定支出: 115,592
差引    : 116,324
```

この時点では、`recur=monthly` / `recur=cycle` の自動展開はしない。

運用ルール:

```text
次サイクルを見たいなら、plan.tsv に次サイクル内の予定を明示する。
```

BQN 側は、書かれた予定を集計するだけにする。

---

## E. 体感とのズレ確認ビュー

`plan.tsv` の予定額が現実的かを見る。

例:

```text
【見込み確認】
光熱費
  前サイクル実績 : 28,500
  次サイクル予定 : 30,000
  差額           : +1,500
  メモ           : 2ヶ月見込 / estimate

住居セット
  前サイクル実績 : 92,500
  次サイクル予定 : 94,000
  差額           : +1,500
```

目的:

- 光熱費の見込みが低すぎないか
- 家賃+光熱費のセットで体感に合っているか
- 「固定費が軽くなる」という見方が過大評価でないか

## BQNでシンプルに保つ方針

今回の相談から見えてきた重要な方針:

```text
BQN側で未来を推測しすぎない。
```

BQNで複雑になるなら、やろうとしていることが雑になっている可能性があります。

したがって、最初は以下を守る。

### 1. 自動展開しない

`recur=monthly` を読んで未来行を生成しない。

`recur` / `series` は、まず人間が予定を見分けるためのメモ。

### 2. 推測しない

「毎月あるはず」「偶数月のはず」をコードが勝手に補わない。

必要な予定は `plan.tsv` に明示する。

### 3. 書かれた予定だけ集計する

BQN は、`plan.tsv` の日付範囲内に書かれた行を合計する。

ただし、次サイクル終了を決める次回収入予定が `plan.tsv` に無い場合は、広い未来範囲を仮定して集計しない。
その場合は「相談材料が未完成」として warning を出す。

これならシンプルで、家計相談でも「予定表のどこが足りないか」を確認しやすい。

### 4. レポートの問いを絞る

最初から全部作らない。

優先順位:

1. 収入日配分ビュー
2. 次サイクル予定ビュー（登録済みベース）
3. 貯金/取り崩しフロー
4. 見込み確認ビュー
5. サブスク変化確認ビュー

## 実装案

## 現在のモジュール境界

今回追加した機能は、現時点では家計簿本体へ深く食い込ませず、**相談用の事実セットを表示する薄い section** として実装している。

### `tools/plan-view.bqn`

分離度は高い。

- read-only
- `plan.tsv` を見やすく表示するだけ
- TSVを書き換えない
- 本体レポートの計算に影響しない

### `cycle-consult`

現時点では `report_sections.bqn` 内の `Sec11` として実装している。

良い点:

- `core.bqn` は変更していない
- `report_engine.bqn` の公開フィールドは増やしていない
- `journal.tsv` / `plan.tsv` の先頭5列は変更していない
- `recur` を自動展開せず、AI判断ロジックも入れていない
- BQN側は「相談用の事実セット」を出し、判断は人間/pit側に残している

注意点:

- `report_sections.bqn` の中で、`journal_rows` / `plan_rows` の日付範囲集計を少し行っている
- 本来の方針では「計算は `report_engine` / 補助モジュール、表示は `report_sections`」なので、育てすぎると renderer が太る

現時点の評価:

```text
plan-view      : 分離されている
cycle-consult  : 軽く食い込んでいるが、試作/最小実装としては許容範囲
core/accounting: 汚していない
```

### 分離するタイミング

`cycle-consult` に次のような機能を追加し始めるなら、`report_sections.bqn` から計算部分を切り出す。

- 貯金/取り崩しフロー
- 前サイクル vs 次サイクル差分
- サブスク棚卸し
- 住居セット表示
- 生活費シナリオ
- 予定カテゴリ別の比較やランキング

切り出し候補:

```text
report_cycle_consult.bqn
```

想定責務:

- `plan.tsv` の日付範囲集計
- 現在サイクルの記録済み実績と残り登録済み予定
- 次サイクル予定
- 収入日当日支払い
- 次回収入予定が無い場合の安全ガード

`report_sections.bqn` は、切り出された値を受け取って表示するだけに戻す。

### 境界の合言葉

```text
BQN本体にAI判断を入れない。
BQN側は相談用の事実セットを出す。
pit/人間がそれを読んで判断する。
```

### Phase 1: `cycle-consult` セクション

既存の `cycle` と `planned` を組み合わせた、人間向け相談ビュー。

表示するもの:

- 現在サイクルの途中経過（`journal.tsv` に記録済み、かつ `cycle_start <= date <= as_of` の実績）
- その差引がサイクル終了時の残額ではなく、基準日以降の支出・未入力取引で減ることの注記
- 現在サイクル残り期間の登録済み予定（`as_of` 〜 `<cycle_end`、`plan.tsv` に明示された予定のみ）
- 次サイクル登録済み予定収入（`cycle_end` 〜 `<その次の収入日`）
- 次サイクル登録済み予定支出
- 登録済み予定ベースの差引
- 収入日当日の予定支出

この Phase では `recur` の自動展開はしない。未来の「着地見込み」は推測せず、記録済み・登録済みの事実セットとして出す。

実装状況:

- [x] `bqn main.bqn --section cycle-consult` を追加
- [x] `plan.tsv` に明示された予定だけを日付範囲で集計
- [x] 次サイクル終了は `plan.tsv` の `cycle_end` より後の次の `income:*` 予定日で判定
- [x] 次回収入予定が無い場合は、開いた未来範囲を集計せず warning を表示
- [ ] 大きい支出 Top は既存 `cycle` セクションを見る運用で、まだ `cycle-consult` には統合しない

### Phase 2: 貯金/取り崩しフロー

対象口座:

- `assets:ゆうちょ`
- `assets:オルカン積立`

サイクル内の資金移動を見て、入金/取り崩し/純増減を出す。

### Phase 3: 住居セット/見込み確認

まずは固定ロジックでよい。

- `expenses:家賃`
- `expenses:光熱費`

を住居セットとして表示する。

将来必要なら `group=housing` メタを追加する。

### Phase 4: サブスク変化確認

まずはカテゴリ単位で十分。

- `expenses:AIサブスク`
- `expenses:サブスク`
- 必要なら `series=` で細分化

## 実装しないこと（当面）

- `recur=monthly` の自動展開
- `recur=cycle` の自動生成
- 貯金額の自動決定
- 100,000円ラインの主役化
- 複雑なルールエンジン化

これらは便利そうだが、今の相談にはまだ早い。

## このレポートで答えたい問い

- 前サイクルで、何が貯金取り崩しにつながったか？
- 次サイクルで、収入直後に何を確保すべきか？
- 次サイクルの予定支出は現実的か？
- 光熱費などの見込みは低すぎないか？
- サブスク解約の効果は予定に反映されているか？
- どれくらい残せば、取り崩しなしで終えられそうか？

この問いに答えられれば、pitとの家計相談は「貯金目標」ではなく、毎サイクルの運用改善として進められる。
